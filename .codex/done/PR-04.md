PR-04: Cache + Registry semantics (Id → Profile rehydration)
Goal

Make I18nId actually useful as a session token: cheap lookups + safe caching.

Scope
crates/i18n-core

Add an internal registry:

I18nId -> Arc<I18nProfile> + optional fallback_chain

Provide API:

resolve_and_cache(req) -> I18nResolution

get(id) -> Option<Arc<I18nProfile>>

insert(profile) -> I18nId (computes canonical CBOR + id)

Cache policy:

simple LRU or size-bounded map (configurable)

expose I18nCacheConfig { max_entries }

Tests

cache hit/miss tests

ensures cached profile matches id (recompute digest from profile and compare)

Acceptance

Other repos can store only I18nId but still get the full profile quickly

PR-04 question (eviction hooks/events)

Keep eviction internal for now.

Reasons:
- If session code depends on eviction events, you’ve turned a cache into a distributed lifecycle problem.
- The correct external behavior is “cache miss → re-resolve”; with I18nId you can deterministically rehydrate the profile if you have either the canonical CBOR or sufficient request inputs.

Recommendation:
- Don’t introduce eviction hooks in PR-04. If needed later, expose helper(s) such as `get_or_resolve(id, hint_req?)` or cache stats (hits/misses/evictions) for observability rather than control-flow hooks.
