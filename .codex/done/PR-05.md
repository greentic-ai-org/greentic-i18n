PR-05: Formatting Facade v1.1 (deterministic now, ICU4X-ready later)
Goal

Keep formatting stable + deterministic today, but define the seam so ICU4X can drop in without breaking APIs.

Scope
crates/i18n-format

Introduce backend trait:

pub trait FormatBackend {
  fn format_number(&self, profile: &I18nProfile, n: DecimalLike) -> String;
  fn format_currency(&self, profile: &I18nProfile, amount: DecimalLike) -> String;
  fn format_datetime(&self, profile: &I18nProfile, instant: SystemTime) -> String;
}


Current implementation becomes BasicBackend

FormatFacade delegates to BasicBackend by default

Define deterministic rules:

decimal separator and grouping exactly as profile states

currency symbol placement rules (explicit and tested)

Add “ICU4X placeholder feature flag”:

--features icu4x compiles stub backend (not necessarily full ICU yet)

Tests

golden formatting tests for a few profiles:

en-GB, fr-FR, ar-OM (even if Arabic numerals are not fully correct yet, keep deterministic output)

Acceptance

Formatting output is deterministic

Backend seam exists so ICU4X is a drop-in later

PR-05 question (BasicBackend vs ICU4X default)

Keep `BasicBackend` permanently available as a deterministic baseline, a fallback for constrained builds, and a known-good output reference for regression tests.

When ICU4X arrives:
- make it optional (`--features icu4x`) while keeping `BasicBackend` as the default to minimize churn.
- later (major/minor bump) you can flip the default to ICU4X but keep `BasicBackend` for CI golden tests, portable builds, and debugging.

Recommendation: PR-05 should make ICU4X opt-in while `BasicBackend` stays the default; plan a separate “default backend switch” PR when the ICU4X path is stable.
