PR-02: Canonical CBOR + I18nId v1 Spec (hash over resolved profile)
Goal

Make I18nId normative + stable by hashing a fully resolved I18nProfile using a canonical CBOR encoding.

Scope
crates/i18n-core

Add a canonical CBOR encoder for I18nProfile:

stable map key ordering

stable omission rules for None

stable enum encoding (string or small int — pick one and freeze it)

stable float encoding rules (avoid floats if possible; use integers / decimal strings for money/units)

Define exact digest pipeline:

digest = blake3(canonical_cbor(profile))

I18nId = "i18n:v1:" + base32(digest[0..16]) (128-bit)

Add:

I18nId::parse(), Display, FromStr

I18nId::version() / I18nId::bytes()

Add doc/spec:

docs/i18n-id-v1.md (normative: encoding rules, fields, version bump rules)

Tests

Golden fixtures:

store fixtures/i18n_id_v1_cases.json:

request inputs → resolved profile canonical CBOR hex → expected I18nId

“Stability guards”:

cargo test fails if any expected ID changes

Acceptance

Same resolved profile always produces same I18nId

Canonical CBOR output is stable across runs/platforms

I18nId parsing rejects malformed IDs

Notes specifically for “Option B / CBOR-only”

In PR-02, treat canonical CBOR encoding as normative (it becomes part of your ABI contract mindset).

Avoid floats in the profile representation; if you need decimals, use integer minor-units or decimal strings so CBOR encoding stays stable. 
 
PR-02 question (floats / decimal strings / minor units)
 
For the current I18nProfile shape, avoid floats entirely in the canonical CBOR payload.
 
Because I18nProfile currently carries only formatting metadata (language/region/script, direction, calendar, timezone ID, currency code, numbering system, separators/grouping, measurement system hints, etc.), there is no need to represent real-valued amounts—so the CBOR profile can stay float-free.
 
If fields that could become numeric/real-valued ever appear (unit scale factors, rounding offsets, etc.), they must be encoded as integers or decimal strings in the profile rather than raw floats.
 
If you truly require a float, freeze canonicalization rules (e.g., reject NaN/±Inf, encode via IEEE-754 float64 only, forbid float32). However, the recommendation is to forbid floats in I18nProfile v1 altogether.
 
Recommendation: add a normative rule in PR-02:
 
“I18nProfile v1 canonical CBOR MUST NOT contain floating-point values; any numeric quantities MUST be encoded as integers or decimal strings.”
 
That keeps I18nId stable and future-proof.
