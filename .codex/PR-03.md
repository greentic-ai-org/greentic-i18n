PR-03: Resolver Contract v1 (precedence + deterministic derivations + fallback chain)
Goal

Turn the resolver into a platform contract (deterministic rules, explicit precedence, fallback chain).

Scope
crates/i18n-core

Define I18nResolution:

pub struct I18nResolution {
  pub id: I18nId,
  pub profile: I18nProfile,
  pub fallback_chain: Vec<I18nTag>,
}


Make precedence explicit and tested:

tenant defaults < user prefs < session overrides < content constraints

Define deterministic derivation rules:

direction (derived unless overridden)

separators/grouping defaults (derived unless overridden)

first_day/hour_cycle (derived unless overridden)

Define error strategy (deterministic):

unknown tz/calendar/numbering system:

either Err (strict) or fallback to known default (per ResolveMode)

Add ResolveMode { Strict, Lenient }

Tests

Precedence tests (table-driven)

Derivation tests (same input → same derived values)

Fallback chain tests: e.g. en-GB → en → default

Acceptance

Resolver behavior is deterministic and documented

I18nId is generated from the resolved profile produced by these rules

PR-03 question (Lenient defaults table)

Keep the Lenient fallback policy small and deterministic rather than building a huge “known defaults” table in v1.

- Calendar: default to gregory.
- Numbering system: default to latn.
- Timezone: prefer the request/tenant timezone when supplied; otherwise fall back to a stable constant such as UTC.
- Direction: derive from language/script; if unknown, default to Ltr.
- First day / hour cycle: use a minimal rule set keyed by a few common regions (e.g., GB/US/SA) with sensible defaults (Monday + h23); otherwise default to Mon/h23.

Recommendation: document this minimal deterministic policy in PR-03 so Lenient mode stays stable; future ICU4X work can improve correctness without breaking the contract by only tightening derivations when stronger inputs exist.
